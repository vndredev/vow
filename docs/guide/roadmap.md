# Roadmap

vow's goal: a complete **dashboard / planning-system app**, operable by a person and an LLM alike. It's built piece by piece вҖ” each element green, proven, and documented before the next.

## Two ways to work with vow

Same truth, same guarantee вҖ” two interfaces:

- **Free** вҖ” the LLM reads one `llms.txt` (the whole concept), then writes `.vow.md` directly. Works with any model, no tooling. The core secures it: parse + the gate + tsgo catch invalid input with **self-healing errors**, so the LLM corrects until green.
- **MCP** вҖ” a typed toolbox (`createEntity`, `addField`, вҖҰ) for token-efficient, direct work, validated _before_ writing.

Both write the same vows. The **core** is the single guarantee (errors вҶ’ 0), regardless of the path вҖ” the safety lives in the core, not in the interface.

## The reference product вҖ” a planning dashboard

The driving goal: a **GitHub-Projects-class planning board**, generated by vow and operable by a human (clicks) and an LLM (operates the spec). It's the example that hardens the generator вҖ” a real, demanding target surfaces exactly which capabilities vow needs.

The key idea: **two kinds of status.** GitHub Projects keeps one hand-maintained status field that drifts. vow splits them:

- **Derived status** (the truth) вҖ” `proposed вҶ’ building вҶ’ proving вҶ’ done / blocked`, read off `fulfills` + `proof` + git + CI + coverage. Never stored, can't lie. Read-only on the board; cards move themselves as work lands.
- **Intentional status** (the plan) вҖ” `priority`, `iteration`, `assignee`, `labels`, order. Set by a human/agent. Editable, draggable; a drag writes back into the vow.

Group a view by a derived field вҶ’ an observability board; group by an intentional field вҶ’ a draggable Kanban that writes back.

### Build order

The critical path вҖ” each element green before the next:

1. **`date`** вҖ” iterations, due dates В· **done**
2. **`reference` + relations** вҖ” item вҶ’ epic, item вҶ’ iteration
3. **`rollup`** вҖ” the derived lifecycle (the heart); status bubbles vow вҶ’ epic вҶ’ roadmap
4. **board / table / roadmap views** вҖ” group-by В· sort В· filter В· drag
5. **observability adapter** вҖ” read git + coverage (+ CI) behind the data-adapter seam
6. **GitHub adapter** вҖ” sync issues / PRs / CI status; manual trigger first, webhook later

The dogfood moment: once the board exists, **its first content is the plan for itself** вҖ” vow planning its own build, in vow.
