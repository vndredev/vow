# Roadmap

vow's goal: build a complete **dashboard / planning-system app** with the core product тАФ operable by a human _and_ an LLM. We get there **slowly**, element by element, function by function; each element ships green and with its own doc page.

## Two ways to work with vow

Same truth, same guarantee тАФ two interfaces:

- **Free** тАФ the LLM reads one `llms.txt` (the whole concept), then writes `.vow.md` directly. Works with any model, no tooling. The core secures it: parse + the gate + tsgo catch invalid input with **self-healing errors**, so the LLM corrects until green.
- **MCP** тАФ a typed toolbox (`createEntity`, `addField`, тАж) for token-efficient, direct work, validated _before_ writing.

Both write the same vows. The **core** is the single guarantee (errors тЖТ 0), regardless of the path тАФ the safety lives in the core, not in the interface.

## The reference product тАФ a planning dashboard

The driving goal: a **GitHub-Projects-class planning board**, generated by vow and operable by a human (clicks) and an LLM (operates the spec). It's the example that hardens the generator тАФ a real, demanding target surfaces exactly which capabilities vow needs.

The key idea: **two kinds of status.** GitHub Projects keeps one hand-maintained status field that drifts. vow splits them:

- **Derived status** (the truth) тАФ `proposed тЖТ building тЖТ proving тЖТ done / blocked`, read off `fulfills` + `proof` + git + CI + coverage. Never stored, can't lie. Read-only on the board; cards move themselves as work lands.
- **Intentional status** (the plan) тАФ `priority`, `iteration`, `assignee`, `labels`, order. Set by a human/agent. Editable, draggable; a drag writes back into the vow.

Group a view by a derived field тЖТ an observability board; group by an intentional field тЖТ a draggable Kanban that writes back.

### Build order

The critical path тАФ each element green before the next:

1. **`date`** тАФ iterations, due dates ┬╖ **done**
2. **`reference` + relations** тАФ item тЖТ epic, item тЖТ iteration
3. **`rollup`** тАФ the derived lifecycle (the heart); status bubbles vow тЖТ epic тЖТ roadmap
4. **board / table / roadmap views** тАФ group-by ┬╖ sort ┬╖ filter ┬╖ drag
5. **observability adapter** тАФ read git + coverage (+ CI) behind the data-adapter seam
6. **GitHub adapter** тАФ sync issues / PRs / CI status; manual trigger first, webhook later

The dogfood moment: once the board exists, **its first content is the plan for itself** тАФ vow planning its own build, in vow.
