# Roadmap

vow's north star: a **GitHub-Projects-class planning dashboard**, generated by vow and operable by a **human** (clicks) and an **LLM** (the spec). It's the demanding target that hardens the generator — and the dogfood moment: once the board exists, its first content is **the plan for itself**, vow planning its own build, in vow.

Built piece by piece — each element green, proven, and documented before the next.

## The foundation (today)

The core mechanism works end-to-end:

- **The vow** — one recursive primitive; `emit entity` / `view` / `form` + `bind`; status is **derived**, never stored.
- **Proof** — the scenario-coverage gate: every promised scenario needs a green test, or the build is red.
- **Generation** — per-entity **zod** schemas; field types (`text` · `longtext` · `number` · `boolean` · `select` · `date` · `reference`) + a stable auto-id; the `## view` vocabulary (semantic blocks + **10 primitives** + 5 layout primitives + `link:` / `icon:` / `model:`); `## form` (validated, per-field errors); multi-view routing.
- **The lists** — referent-display (a `reference` shows the target's **name**, not its id) + status **badges** (a `select` renders as a chip).
- **Runtime + chrome** — `@vow/store` (in-memory), `@vow/router`, `@vow/shell` (the dashboard chrome: sidebar · dark mode · mobile drawer), `@vow/theme` (swappable tokens), `@vow/icons`.
- **The docs** — a generated vow app, dogfooding vow's own primitives in prose; a generated `/llms.txt` for an LLM.
- **One model, many frameworks** — the component IR is framework-free; Vue is the first adapter, React/Solid are more adapters over the same core.

## The path to the dashboard

**Phase 1 — Data-display primitives.** `Table` (header · rows · cells, then sort/filter) · `Card` · `Stats` / metric · `Callout`/alert. The entity list becomes a real **Table**. → a read-only observability view over your entities.

**Phase 2 — View patterns + slicing.** The `## view` vocabulary gains `table` · `cards` · `board` · `detail` · `stats`; **group-by · sort · filter** on a view. → slice the data (group tasks by status, filter by assignee).

**Phase 3 — The board (the interactive core).** Kanban — group-by a field → lanes; **drag** a card → writes back into the vow. The **two kinds of status** (below) come alive: derived = read-only, intentional = draggable.

**Phase 4 — Real data + observability.** in-memory → **Cloudflare D1** (real persistence) behind the `useCollection` seam; the **observability adapter** (read git + coverage + CI → the derived status); the **GitHub adapter** (sync issues / PRs / CI). → the board reflects real work, live.

**Phase 5 — The author layer (LLM-first).** `serialize` (Vow → vow.md) · a typed **mutation API** (`addEntity` / `addField` / …) · the **vow MCP server** (typed tools over the mutation API, validated _before_ writing). → the LLM operates vow directly.

## The two kinds of status

GitHub Projects keeps one hand-maintained status field that drifts. vow splits it:

- **Derived status** (the truth) — `proposed → building → proving → done / blocked`, read off `fulfills` + `proof` + git + CI + coverage. Never stored, can't lie. Read-only; cards move themselves as work lands.
- **Intentional status** (the plan) — `priority`, `iteration`, `assignee`, `labels`, order. Set by a human or an agent. Editable, draggable; a drag writes back into the vow.

Group a view by a derived field → an observability board; by an intentional field → a draggable Kanban that writes back. The `rollup` (Phase 3–4) bubbles derived status **vow → epic → roadmap**.

## Supporting work (alongside the phases)

The UI primitives the patterns pull in — `Callout`/`Alert` · `Avatar` · `Tooltip` · `Menu`/dropdown · `Toast` · `Pagination` · `Breadcrumb` · `Empty-state` — plus **multi-value fields** (labels, assignees), **field-presentation control** (a boolean as a switch, a select as a radio), a **value→variant** map for status badges (`done` → success), referent-display **in forms**, and more icon libraries (Heroicons / Phosphor) + config-selection.

## Two ways to work

Same truth, same guarantee — two interfaces. The **core** is the single guarantee (invalid input → 0, self-healing), regardless of the path:

- **Free** — the LLM reads one `/llms.txt`, then writes `.vow.md` directly. Any model, no tooling.
- **MCP** — a typed toolbox (`createEntity`, `addField`, …), token-efficient, validated _before_ writing.
