# Roadmap

vow's goal: build a complete **dashboard / planning-system app** with the core product ‚Äî operable by a human _and_ an LLM. We get there **slowly**, element by element, function by function; each element ships green and with its own doc page.

## Two ways to work with vow

Same truth, same guarantee ‚Äî two interfaces:

- **Free** ‚Äî the LLM reads one `llms.txt` (the whole concept), then writes `.vow.md` directly. Works with any model, no tooling. The core secures it: parse + the gate + tsgo catch invalid input with **self-healing errors**, so the LLM corrects until green.
- **MCP** ‚Äî a typed toolbox (`createEntity`, `addField`, ‚Ä¶) for token-efficient, direct work, validated _before_ writing.

Both write the same vows. The **core** is the single guarantee (errors ‚Üí 0), regardless of the path ‚Äî the safety lives in the core, not in the interface.

## The reference product ‚Äî a planning dashboard

The driving goal: a **GitHub-Projects-class planning board**, generated by vow and operable by a human (clicks) and an LLM (operates the spec). It's the example that hardens the generator ‚Äî a real, demanding target surfaces exactly which capabilities vow needs.

The key idea: **two kinds of status.** GitHub Projects keeps one hand-maintained status field that drifts. vow splits them:

- **Derived status** (the truth) ‚Äî `proposed ‚Üí building ‚Üí proving ‚Üí done / blocked`, read off `fulfills` + `proof` + git + CI + coverage. Never stored, can't lie. Read-only on the board; cards move themselves as work lands.
- **Intentional status** (the plan) ‚Äî `priority`, `iteration`, `assignee`, `labels`, order. Set by a human/agent. Editable, draggable; a drag writes back into the vow.

Group a view by a derived field ‚Üí an observability board; group by an intentional field ‚Üí a draggable Kanban that writes back.

### Build order

The critical path ‚Äî each element green before the next:

1. **`date`** ‚Äî iterations, due dates ¬∑ **done**
2. **`reference` + relations** ‚Äî item ‚Üí epic, item ‚Üí iteration
3. **`rollup`** ‚Äî the derived lifecycle (the heart); status bubbles vow ‚Üí epic ‚Üí roadmap
4. **board / table / roadmap views** ‚Äî group-by ¬∑ sort ¬∑ filter ¬∑ drag
5. **observability adapter** ‚Äî read git + coverage (+ CI) behind the data-adapter seam
6. **GitHub adapter** ‚Äî sync issues / PRs / CI status; manual trigger first, webhook later

The dogfood moment: once the board exists, **its first content is the plan for itself** ‚Äî vow planning its own build, in vow.
